CHAPITRE X : MIGRATION POSTGRESQL ET INTÉGRATION FRONT-END (PHASE 5)


1. Introduction et Contexte de la Phase 5

L'application Baana (by SEN DIGITAL PULSE) a franchi une étape décisive lors de cette cinquième phase de développement. Après avoir conçu l'architecture logicielle de base, défini la logique métier complexe et développé l'interface de l'application mobile en Flutter (réalisée en collaboration avec le pôle Front-end "Héritage Signature Premium"), il était impératif de connecter et de synchroniser les deux environnements de manière sécurisée.

Mon objectif principal et ma responsabilité en tant qu'Architecte Backend étaient d'abandonner définitivement la base de données légère (SQLite) utilisée jusqu'ici pour les tests de développement locaux, et de la remplacer par un Système de Gestion de Base de Données Relationnelle (SGBDR) extrêmement robuste, hautement disponible et taillé pour la mise en production : PostgreSQL. Cette transition était un prérequis indispensable pour garantir l'intégrité absolue des données financières, sécuriser les transactions e-commerce B2B et supporter une charge importante de commerçants simultanés.




2. Sécurisation des Accès : Implémentation du standard JWT

L'application mobile (Flutter) nécessitant une communication avec une API de type "Stateless" (c'est-à-dire sans aucun état ni session mémorisée sur le serveur), j'ai dû concevoir et mettre en place un système d'authentification cryptographique basé sur la norme internationale JSON Web Tokens (JWT).

Le processus est le suivant : lorsqu'un utilisateur vérifie avec succès son code OTP (One-Time Password) reçu par SMS, le backend génère un token crypté via l'algorithme HMAC SHA-256. Ce token contient, de manière sécurisée, l'identifiant du commerçant ainsi que son statut d'abonnement (ex: isPro). Ce token agit dès lors comme un passeport numérique infalsifiable, garantissant la sécurité de toutes les requêtes ultérieures vers l'API sans avoir besoin d'interroger inutilement la base de données à chaque vérification de route.

3. Initialisation du Catalogue Sénégalais (Seeders)

Pour que mon binôme en charge du Front-end puisse tester son application mobile dans des conditions strictement réelles, j'ai développé un script d'initialisation automatisé (init_seed.js). Ce script a pour mission de peupler massivement la base de données PostgreSQL avec un catalogue de produits e-commerce 100% adapté aux réalités du marché local sénégalais (ex: Huile d'arachide en bidon, Sac de Riz brisé parfumé, Bouillons, etc.).

Outre les produits, ce script génère l'architecture des catégories (Alimentaire, Cosmétique, Entretien) et intègre un compte utilisateur de test possédant les privilèges "Abonné Pro". Cela nous a permis de valider avec précision l'application des réductions exclusives sur les prix de gros.

4. Gestion des Commandes et Intégrité Transactionnelle

Le cœur névralgique de tout système e-commerce réside incontestablement dans son tunnel de paiement et de validation des achats (Checkout). J'ai implémenté l'endpoint de création de commande (POST /api/orders) en exploitant le puissant mécanisme des Transactions SQL offert nativement par PostgreSQL.

Cette approche architecturale est vitale car elle garantit le respect strict du principe ACID (Atomicité, Cohérence, Isolation, Durabilité). Si la moindre erreur survient lors du processus (par exemple lors de la déduction du stock d'un produit ou lors de l'enregistrement du statut de paiement WAVE), l'ensemble de la transaction est immédiatement annulé (Rollback automatique). Cela empêche formellement la création de commandes fantômes ou l'apparition d'incohérences de stock critiques. Le système algorithmique vérifie également dynamiquement si l'utilisateur est un abonné Pro afin de substituer le prix public par le tarif réduit (proPrice).

5. Algorithmique Avancée : Le Dashboard Analytique

L'une des propositions de valeur fondamentales de l'application Baana est sa capacité à démontrer tangiblement aux commerçants la rentabilité financière de leur abonnement Pro. À cette fin, j'ai conçu un contrôleur analytique (dashboardController.js) dont le rôle est d'effectuer des calculs d'agrégation mathématique complexes directement au niveau du backend.

L'endpoint de statistiques renvoie en temps réel des "Key Performance Indicators" (KPIs) cruciaux pour l'utilisateur : le total global dépensé sur la plateforme, le nombre de livraisons gratuites restantes liées à l'abonnement, et surtout, le montant exact des économies réalisées calculé via le différentiel cumulé entre le prix public de chaque article acheté et son prix de gros.

6. Conclusion de la Phase 5

Cette cinquième phase marque l'aboutissement fonctionnel du développement Backend de l'application Baana. La migration technologique réussie vers PostgreSQL, couplée à la sécurisation stricte des endpoints via JWT et à la gestion transactionnelle rigoureuse des processus d'achat, a permis de fournir au pôle Front-end une API RESTful parfaitement documentée, hautement stable et prête pour une utilisation intensive.

L'application mobile Flutter étant désormais entièrement synchronisée avec les données réelles du serveur, la voie est libre pour entamer la sixième et dernière phase de notre projet de fin d'étude : le Déploiement Cloud et l'Hébergement en production.
